# 脚本安全规范

## 目录结构

```
scripts/
├── dev/          # 项目生命周期（生产必需）
├── seed/         # 测试数据（仅开发环境）
├── batch/        # 批量处理（仅开发环境）
└── sync/         # 数据同步（仅开发环境）
```

## 环境限制

### 生产环境禁止运行的脚本

| 目录 | 原因 |
|------|------|
| `seed/` | 生成假数据，污染生产数据库 |
| `batch/` | 绕过权限验证，直接操作数据库 |
| `sync/` | 外部数据直接入库，无审计 |

### 安全措施

**方案一：环境变量检查（推荐）**

在每个脚本开头添加：

```typescript
// 生产环境禁止运行
if (process.env.NODE_ENV === 'production') {
  console.error('❌ 此脚本禁止在生产环境运行');
  process.exit(1);
}
```

**方案二：物理隔离**

```
scripts/
├── dev/              # 生产可用
└── internal/         # 仅开发环境
    ├── seed/
    ├── batch/
    └── sync/
```

部署时排除 `internal/` 目录：

```bash
# .dockerignore 或部署脚本
rm -rf scripts/internal
```

## 运行规范

| 脚本类型 | 运行方式 | 审批要求 |
|----------|----------|----------|
| dev/* | 自动化部署 | 无需审批 |
| seed/* | 开发者本地运行 | 无需审批 |
| batch/* | 需评审后运行 | 需 Tech Lead 审批 |
| sync/* | 需评审后运行 | 需 Tech Lead 审批 |

## 新增脚本要求

新增脚本必须：

1. **添加环境检查**：禁止在生产环境运行
2. **添加使用说明**：用途、风险、回滚方案
3. **添加审批流程**：谁批准才能运行
